# Codex 插件逐个适配审计

生成日期：2026-06-20
更新日期：2026-06-21

## 说明

这份文档按当前本机 Codex 插件市场、插件清单文件、本机缓存和当前会话可见工具，对 178 个插件逐个做个人工作流适配判断。判断对象不是“插件有没有名气”，而是它是否能服务 SignalProof 的信号、调研、辩论、验证、产物、发布、反馈、决策、资产化和流程自检。

重要边界：

- 插件市场可见不等于已经安装。
- 已安装不等于已经授权外部账号。
- 有工具入口不等于证据合格。每次真实调用仍要写入具体案例的 `tool-ledger.md`。
- 未授权、额度不足、接口失败要写成 `blocked/auth`、`blocked/quota` 或 `failed/tool`，不能当成 0 结果。

本轮依据：

- `codex plugin list`：当前插件市场安装状态。
- 插件清单文件：类别、描述、能力形态和授权策略。
- `/Users/rust/.codex/plugins/cache`：本机缓存和运行时插件。
- 当前会话工具：Computer Use 已实测成功但插件市场当前显示未安装；Record & Replay 的 `event_stream_status` 已实测可见但插件市场当前显示未安装；Browser 上轮实测失败。
- 最新 Codex 官方手册：插件可打包技能、应用集成和 MCP 服务器。

## SignalProof 阶段映射

- **信号**：`browser`、`chrome`、`github`、`record-and-replay`。信号来自网页、登录态页面、开源仓库或用户演示时使用。
- **调研**：`readwise`、`zotero`、`scite`、`semrush`、`similarweb`、`brand24`、`github`、`hugging-face`、`google-drive`、`notion`。补真实讨论、论文、竞品、流量、历史资料和技术证据。
- **辩论**：`scite`、`github`、`browser`、`documents`、`pdf`。找反证、替代方案、失败案例和关键风险。
- **验证**：`browser`、`chrome`、`computer-use`、`posthog`、`mixpanel`、`amplitude`、`sentry`、`datadog`、`codex-security`。验证页面、行为、运行态、错误和安全。
- **产物**：`documents`、`pdf`、`spreadsheets`、`presentations`、`build-web-data-visualization`、`hyperframes`、`canva`、`figma`。生成非 Markdown 资产并验收。
- **发布**：`github`、`vercel`、`netlify`、`cloudflare`、`browser`、`chrome`、`gmail`、`slack`。确认公开入口和触达渠道。
- **反馈**：`brand24`、`dovetail`、`posthog`、`mixpanel`、`amplitude`、`github`、`gmail`、`slack`、`jam`。回收真实讨论、用户原话、行为指标和公开问题。
- **资产**：`record-and-replay`、`documents`、`pdf`、`spreadsheets`、`presentations`、`notion`、`mem`、`google-drive`、`airtable`。把有效流程、报告、表格、演示和知识库沉淀为复利资产。
- **流程自检**：`computer-use`、`browser`、`record-and-replay`、`documents`、`pdf`、`spreadsheets`、`presentations`。证明页面、文件、应用和流程真实可跑。

## 使用原则

1. 先按阶段选候选插件，不按清单机械全装。
2. A 级插件也要先验证授权、数据覆盖和结果质量。
3. B 级插件通常要等到有真实账号、真实数据源或具体案例后再试。
4. C 级插件保留为专项备用，不进入默认流程。
5. D 级插件暂不纳入，除非出现完全匹配的垂直案例。
6. 每次插件调用都要写入 `tool-ledger.md`：候选、实际调用、原始查询/对象、结果强度、失败原因、下一步。
