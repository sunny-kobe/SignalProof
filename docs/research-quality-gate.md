# 研究准确性质量门

这份规则回答一个问题：研究结论到底有多确定，能不能继续往后走。

SignalProof 里的研究阶段不以“调用过工具”为通过标准，而以来源覆盖、交叉验证、反证处理和结论许可为通过标准。工具调用只是一条线索，不能自动变成证据。

## 1. 最少来源覆盖

每个真实机会研究至少考虑 4 类来源，并在 `research.md` 和 `tool-ledger.md` 写清楚是否覆盖：

| 来源类型 | 目标 | 常用能力 | 通过标准 |
| --- | --- | --- | --- |
| 公开讨论 | 找到目标人群原话、抱怨、采用、反对意见 | `last30days`、Reddit、HN、X、YouTube、TikTok、Threads、Browser、Chrome | 命中具体人群和具体任务，而不是泛 AI 热闹 |
| 项目和数据 | 验证项目是否真的有人用、维护、付费、访问或讨论 | GitHub、Hugging Face、npm/PyPI、Similarweb、Semrush、PostHog、Sentry | 有可核验指标、时间窗和对象 |
| 官方和一手资料 | 确认 API、限制、价格、政策、技术事实 | 官方文档、OpenAI Docs、GitHub release、论文、PDF、Documents | 来源直接、日期明确、能复查 |
| 反证和替代方案 | 找到“为什么不成立”和“用户现在怎么解决” | Scite、Readwise、Zotero、GitHub issues、竞品文档、搜索 | 至少写出一个强替代或失败风险 |

如果关键来源因为认证、额度、权限、403/402、0 结果、cookie 读取失败而没有覆盖，必须写成来源缺口，不能写成“没有需求”。

## 2. 交叉验证规则

研究结论至少要回答：

- 谁在说：是否命中目标用户，而不是泛泛的开发者或 AI 爱好者。
- 说什么：是否有具体任务、失败场景、付费/迁移/替代动作。
- 在哪里说：来源平台、时间窗、查询词和对象是否可复查。
- 有没有反证：是否存在成熟替代、低意愿、低频、已有工作流或渠道不匹配。
- 能支持什么结论：只能支持继续研究、低成本实验、暂停或放弃，不能直接支持产品化。

## 3. 证据等级

| 等级 | 判定标准 | 允许推进到 |
| --- | --- | --- |
| strong / passed | 至少 3 类来源互相印证，其中包含目标用户原话或可核验指标，并处理过反证 | 可进入验证计划或公开产物；仍不能等同市场验证 |
| medium | 至少 2 类来源有效，结论需要一定推断，反证没有推翻方向 | 低成本内部实验、窄范围公开验证 |
| weak | 只有单一来源、噪音大、命中泛化、关键来源缺失或内部推理为主 | 继续补研究或仅做内部流程实验 |
| blocked | 关键来源被认证、额度、权限、工具失败阻断 | 先处理授权/API/账号或改用降级来源 |

## 4. 结论许可

研究阶段的结论只能落在下面几类：

- `继续研究`：证据太弱，但方向还有必要查。
- `低成本实验`：证据中等或局部强，适合做小产物验证。
- `暂停`：关键来源缺失或外部证据不足，等待授权、渠道或更窄问题。
- `放弃`：反证强、替代成熟、目标人群不痛或验证成本过高。

禁止直接写：

- 已验证需求；
- 市场已验证；
- 可以产品化；
- 可以做 SaaS；
- 插件已安装所以证据成立。

## 5. `research.md` 必填块

每个真实 case 的 `research.md` 至少包含：

- 研究问题；
- 来源覆盖表；
- 交叉验证；
- 反证和替代方案；
- 证据等级；
- 结论许可；
- 需要用户授权或开通；
- 下一步补证。

## 6. `tool-ledger.md` 必填记录

研究阶段至少记录：

- 候选能力；
- 实际调用；
- 查询词、对象、时间窗；
- 结果质量：强 / 中 / 弱 / 失败 / 阻塞；
- 没用的原因；
- 是否改变判断；
- 下一步补跑或替代来源。

## 7. 当前本机授权缺口

截至当前诊断，GitHub 已可用，`last30days` 可运行，OpenRouter provider 可用。当前处理方式：

- Full Disk Access 已验证通过，`last30days --diagnose` 不再报 Safari cookie 权限错误；
- X / xurl OAuth 已验证通过，但 X API 检索返回 `CreditsDepleted`；当前标记为 `暂缺但非阻断`，不作为必备付费项，只在 case 明确依赖 X 实时讨论、早期热度或 KOL 扩散时提醒补 credits；
- Readwise、Scite、Semrush、Similarweb、Brand24 等研究增强插件已由用户网页登录，但当前 Codex 工具面未暴露直接 connector；按真实 case 做只读探针，不要一次性全跑；
- PostHog、Mixpanel、Amplitude、Sentry、Datadog 属于产品运行/反馈阶段，外部反馈暂停时不作为当前必备。

更细的用户授权清单见 [`docs/user-auth-and-api-checklist.md`](user-auth-and-api-checklist.md)。

## 8. 新 case 严格检查

新建或正在维护的 case 完成前，运行：

```bash
python3 scripts/signalproof.py check-case <case-slug> --strict
```

这个命令会把研究质量门 warning 视为失败。它适用于新 case 和主动维护中的 case；旧 case 的历史 warning 可以保留在 `check-all` 输出中，作为后续专项补齐清单。

最小通过期待：

- `research.md` 覆盖来源覆盖、交叉验证、反证和替代方案、证据等级、结论许可、用户授权缺口；
- `tool-ledger.md` 覆盖候选能力、实际使用、结果质量、跳过原因、是否改变判断；
- `feedback.md` 明确真实反馈是否为空；
- `flow-review.md` 记录证据质量、过度声称风险和下一轮优化。

同时，新 case 不能保留未完成占位标记，包括 `TODO`、`TBD`、`待补`、`待定` 和未替换模板变量。外部机会 case 可以在创建初期保留这些标记，但声称完成前必须改成真实来源、明确跳过原因或证据缺口。
