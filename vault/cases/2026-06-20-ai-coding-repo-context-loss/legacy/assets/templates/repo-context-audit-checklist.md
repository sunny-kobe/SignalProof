# Repo AI Context Audit Checklist

> 当前是验证中的 checklist，不是真实成功案例。

## 1. 失败模式

- 这个 repo 里 AI 最常写偏的 3 个地方是什么？
- 最近一次 AI 写偏任务是什么？
- 写偏是因为目录边界、接口契约、组件复用、样式规范、验证命令，还是需求描述不清？
- 这些失败模式是否值得写进 `AGENTS.md` / `CLAUDE.md`？

## 2. 项目形状

- agent 是否知道主要入口文件？
- agent 是否知道哪些目录是业务页面、公共组件、API 层、工具函数、类型定义？
- agent 是否知道哪些目录不能随便改？
- 是否存在子目录级规则，应该放到更近的 `AGENTS.md`？

## 3. 业务契约

- 是否有容易被 AI 改错的枚举值，例如 `0/1`、字符串状态、空值语义？
- 是否有容易被 AI 改错的接口形态，例如数组变对象、对象变数组、分页字段名？
- 是否有必须保留的后端字段、权限码、接口路径或查询参数？
- 是否写清“遇到不确定契约时先确认，不要自造字段”？

## 4. 复用边界

- agent 是否知道已有组件、hook、formatter、请求封装、错误处理方式？
- 是否写清优先复用本项目模式，而不是套通用最佳实践？
- 是否列出最常被重复造轮子的模块？

## 5. 验证方式

- agent 是否知道本 repo 最小验证命令？
- 是否有浏览器验收步骤？
- 是否有 lint、typecheck、unit test 或 smoke test？
- 是否写清哪些失败可以接受，哪些失败必须阻断？

## 6. 上下文瘦身

- 是否删除了“写高质量代码”“保持简洁”这类泛泛规则？
- 是否删除了 README 里已经有、但不会改变 agent 行为的内容？
- 每个区块是否只保留 3 到 7 条关键约束？
- 文件是否短到 agent 能稳定读完和遵守？

## 7. Before / After 验证

- 选择一个真实 AI 写偏任务。
- 先不加新规则，记录 before 输出。
- 用 checklist 更新 `AGENTS.md` / `CLAUDE.md`。
- 重跑同一任务，记录 after 输出。
- 对比是否更符合项目约定。
- 如果没有改善，记录为反证，不要强行解释为成功。
